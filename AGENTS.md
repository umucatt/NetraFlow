# NetraFlow 长期工程规则

本文件只保留跨版本长期有效的工程约束。具体支持平台、版本号、产物名称、数据路径和发布细节，以当前代码、`package.json`、工作流和 README 为准。

## 基本原则

- 只维护一套共享业务源码。
- 平台差异集中在轻量、明确的适配层。
- 不按 Windows、macOS、Linux 复制业务实现。
- 不为抽象而抽象，不为未来假设增加当前复杂度。
- 优先修复根因，避免无关重构。
- 修改前先阅读现有实现、测试和相关文档。
- 不因新增平台支持破坏已经稳定的平台行为。

## 平台边界

- 桌面平台支持范围以当前构建配置和 README 为准。
- 未经明确要求，不增加新的 CPU 架构、包格式或移动端目标。
- 不创建 React Native、Capacitor 或其他移动端工程。
- Windows 保留现有窗口行为和安装、便携、升级、重装、卸载兼容性。
- macOS 使用平台原生窗口控制和正常应用生命周期。
- Linux 使用系统原生窗口控制，并严格处理大小写敏感路径。
- 不通过 CSS 或自定义控件强行模拟完整系统窗口外壳。

## 数据与存储

- 用户数据、运行数据和缓存必须集中、可预测。
- 正式业务数据只写入由统一存储路径模块管理的 `userdata`。
- Electron profile、缓存、日志和临时状态只写入由统一存储路径模块管理的 `runtime`。
- 不把 NetraFlow 数据散落到无关目录。
- 不扫描、修改或删除其他应用数据。
- 不删除用户导出到 NetraFlow 管理目录之外的文件。
- 不处理操作系统自身维护且不属于 NetraFlow 的记录。
- 渲染进程不得提供任意文件系统路径用于删除。
- 删除目标必须由主进程计算并验证。
- 禁止删除用户主目录、磁盘根目录、系统目录、Applications 或其他应用目录。
- 修改数据结构时必须同时考虑 schema、兼容性、拒绝未来版本和恢复策略。
- 未经明确要求，不迁移现有正式用户数据目录。
- 不把开发、测试、示例、缓存、日志或本机路径打入发行包。

## 清除与破坏性操作

- 最高级清除操作必须明确区分于普通退出。
- 清除开始后必须停止新的持久化写入、自动保存、自动快照和后台任务。
- 清除后不得由退出流程重新创建已删除的数据或设置。
- 清除只能作用于 NetraFlow 当前明确管理的数据目录。
- 不依赖下一次启动完成当前清除。
- 不增加隐藏清理脚本或后台清理程序。
- 不为提高删除成功率申请更高权限。
- 用户导出的外部备份不得被清除。

## 权限与外部进程

- 应用运行时只使用当前用户普通权限。
- 不请求管理员、root、sudo、pkexec、辅助功能或自动化控制权限。
- 不修改系统安全设置，不绕过 Gatekeeper、UAC 或其他平台保护机制。
- 产品功能不得依赖 `cmd.exe`、PowerShell、Bash、`sh`、`osascript` 或隐藏脚本。
- 产品功能不得通过 `exec`、`execFile` 或 Shell 命令完成清理、安装、修复或卸载。
- 文件操作优先使用 Electron 官方 API 和 Node.js 文件系统 API。
- 构建、测试和 CI 脚本可使用所在平台正常提供的 Shell，但不得把该能力带入应用运行时。

## 安全与隐私

- 不擅自增加网络请求、遥测、分析统计、后台服务、开机启动、系统托盘常驻或自动更新。
- 不把登录密码、可复用解密凭证或敏感密钥持久化到不应保存的位置。
- 修改加密、密码、快照或完整性逻辑时，必须同步检查兼容性、失败恢复和用户数据风险。
- 不在文档、日志或 Release notes 中声称未实际完成的签名、公证、安全认证或平台验证。

## 依赖与构建

- 有 lockfile 时优先使用 `npm ci`。
- 未经明确需要，不新增或升级依赖。
- 无依赖变化时不得修改 `package-lock.json` 的依赖树、`integrity`、`resolved` 或 `lockfileVersion`。
- 版本更新时，`package.json` 与 `package-lock.json` 中的根版本必须保持一致。
- 打包脚本必须从 `package.json` 动态读取版本，不得硬编码当前版本。
- 正式产物命名、架构和包格式以当前发布配置和契约测试为准。
- 构建产物不得进入 Git 状态。

## 自动化验证与发布

- 普通提交和 Pull Request 只运行验证流程，不创建正式发行包或 Release。
- 正式发布只由版本 tag 触发。
- tag 必须使用 `v<version>`，并指向实际发布 commit。
- 一个版本只对应一个 tag 和一个 GitHub Release。
- 所有平台必须从同一个 tag、同一个 commit、同一个版本号构建。
- 各平台构建任务只上传内部 Actions artifacts。
- 最终只能由一个发布任务创建一个 Draft Release。
- 不允许不同平台分别创建 Release。
- Release 只上传当前定义的正式平台产物。
- GitHub 自动生成的源码归档不得重复上传。
- 校验和写入 Release notes 或内部构建记录，不作为额外 Release asset 上传，除非产品规则明确改变。
- 发布工作流不修改版本号、不创建 tag、不提交代码。
- 未经明确授权，不得触发、修改或执行远端发布操作。

## Git 操作边界

除非用户针对当前操作明确授权，否则不得：

- `git add`
- 创建本地 commit
- 修改已有 commit
- 创建或切换到新分支
- 设置 upstream
- `git push`
- 强制推送
- 创建远端分支
- 创建 Pull Request
- 创建或推送 tag
- 创建 Release
- 上传构建产物
- 修改 GitHub 仓库设置、Secrets 或 Actions 配置
- 使用 GitHub API 写入远端内容

完成代码修改不代表自动获得提交或推送权限。

## 测试与验证

修改后按影响范围运行：

- `npm run typecheck`
- `npm test`
- `npm run build`
- `git diff --check`

涉及发布时还应运行：

- 发布检查
- 产物命名和内容校验
- 对应平台的实际打包验证

平台产物应尽量在目标操作系统或正式 CI runner 上构建和验证。

不得把静态检查、契约测试或非目标平台模拟结果描述为真实平台运行通过。

发现测试、构建或数据安全问题时应停止，并报告明确证据，不得绕过检查继续发布。

## 文档一致性

以下内容发生变化时，应同步检查 README、CHANGELOG、工程规则、工作流、发布脚本和契约测试：

- 支持平台或架构
- 正式产物名称或包格式
- 数据目录或数据格式
- 密码、加密和快照行为
- 构建、验证或发布流程
- 签名、公证或自动更新状态

历史 CHANGELOG 记录不得为了适配当前版本而改写。

## 完成报告

任务完成后至少说明：

- 修改了哪些文件及目的
- 是否影响数据格式、数据目录、窗口行为或构建配置
- 是否新增依赖或修改 lockfile
- 执行了哪些检查及结果
- 当前 Git 状态
- 是否执行暂存、commit、push、tag、Release 或其他远端操作

未执行的远端操作应明确写明未执行。

## 最终原则

> 一套业务代码、轻量平台适配、数据边界清晰、无提权、无隐藏命令、验证后再发布、未经授权不修改 Git 或远端状态
