# NetraFlow 工程规则

## 当前开发目标

NetraFlow 0.9.10 将在现有 0.9.9 Windows 版本基础上扩展桌面平台支持。

当前支持目标：

- Windows x64
  - NSIS 安装版
  - 便携版
- macOS
  - 仅支持 Apple Silicon arm64
  - 使用 DMG 分发
  - 不构建 Intel x64
  - 不构建 Universal
- Ubuntu/Linux
  - x86_64 AppImage
  - 当前只维护一种 Linux 通用分发格式

当前阶段不增加移动端目标。

## 开发与 Git 工作流

- 所有开发工作在本地 dev 目录进行。
- 使用本地 Git 管理代码修改。
- 未经用户针对具体操作明确授权，不得修改任何远端内容。
- 完成任务不代表获得推送权限。
- 不得因为备份、同步或测试需要擅自推送。
- 本地开发期间保持远端 `main` 不变。
- 后续多设备测试应使用同一个 commit SHA。
- 未经允许，不得设置 upstream。

未经明确授权，禁止：

```text
git push
git push --force
git push --set-upstream
创建远端分支
创建 Pull Request
创建或推送 tag
创建 Release
上传构建产物
修改 GitHub Actions
修改 GitHub 仓库设置
使用 GitHub API 写入
```

## 总体架构原则

- 保持一套共享业务代码。
- 只增加轻量、明确的平台适配层。
- 不拆分为三套独立业务代码。
- 不因跨平台适配破坏已经稳定的 Windows 行为。
- 不为了未来可能的移动端迁移为 monorepo。
- 不提前创建移动端工程。
- 纯业务逻辑应尽量避免依赖 Electron。
- IPC 负责传输，不应承担主要业务逻辑。
- 平台判断应集中处理。
- 不在业务代码中散落大量平台判断。
- 优先采用最小修改。
- 不为抽象而抽象。

## Windows 兼容要求

- 保留 Windows x64 NSIS 安装版。
- 保留 Windows x64 便携版。
- 保留现有安装版和便携版数据逻辑。
- 未经批准不得迁移 Windows 用户数据目录。
- 保留 Windows 卸载器中的数据清除功能。
- Windows 继续使用现有无边框和自绘窗口控制。
- macOS 和 Linux 改动不得破坏 Windows。
- 保持升级安装、重新安装、卸载和便携行为兼容。
- 不得仅为跨平台统一删除稳定的 Windows 实现。

## macOS 要求

- 仅支持 Apple Silicon arm64。
- 不增加 Intel x64。
- 不增加 Universal。
- 使用 DMG 分发。
- 不使用 AppImage。
- 当前不使用 PKG。
- 使用 macOS 原生 traffic lights。
- 遵循 macOS 正常应用生命周期。
- 正确处理关闭窗口、Dock 激活和重新创建窗口。
- 不申请管理员权限。
- 不申请 root 权限。
- 不申请辅助功能权限。
- 不申请自动化控制权限。
- 不修改系统安全设置。
- 不绕过 Gatekeeper。

## Ubuntu/Linux 要求

- 当前只构建 x86_64 AppImage。
- 未经要求不增加其他 Linux 包格式。
- 第一版优先使用系统原生窗口控制。
- 按照大小写敏感文件系统处理路径。
- 不依赖 Windows 大小写不敏感行为。
- 不使用 sudo。
- 不使用 pkexec。
- 不请求 root 权限。
- 不执行系统级安装。
- 不向系统目录写入 NF 用户数据。

## 数据目录与克制性

NetraFlow 必须保持克制、集中、可预测。

- 所有 NF 主动管理的数据必须位于一个明确的数据根目录下。
- 正式用户数据统一放在 `userdata`。
- Electron profile、缓存、日志、崩溃数据和临时状态统一放在 `runtime`。
- 不得把 NF 数据散落到无关目录。
- 不扫描无关用户目录或系统目录。
- 不修改或删除其他应用数据。
- 不删除用户导出到 NF 根目录外的文件。
- 不处理操作系统自身维护且不属于 NF 的记录。
- 删除路径必须由主进程计算。
- 渲染进程不得提供任意删除路径。
- 删除前必须验证路径。
- 禁止删除用户主目录、磁盘根目录、系统目录、Applications 或其他应用目录。

Windows 保留现有：

```text
userdata/
runtime/
```

macOS 使用当前用户拥有的集中目录，例如：

```text
~/Library/Application Support/NetraFlow/
  userdata/
  runtime/
```

Ubuntu/Linux 使用当前用户拥有的集中目录，例如：

```text
~/.config/NetraFlow/
  userdata/
  runtime/
```

具体位置由统一存储路径模块管理。

## 清除全部数据

不得为 macOS 或 Linux 单独增加新的清除按钮。

升级现有最高级清除功能，使其成为全平台一致的清除全部本地数据并退出操作。

该操作必须：

- 停止新的持久化写入；
- 停止自动保存；
- 停止自动快照；
- 停止后台任务；
- 清除内存中的密钥和敏感状态；
- 删除全部 NF 管理的 `userdata`；
- 清除 Electron Session 数据；
- 清除缓存；
- 删除全部 NF 管理且当前进程可以删除的 `runtime`；
- 完成后立即关闭 NetraFlow；
- 不依赖下一次启动；
- 不增加下次启动清理机制；
- 假设用户下一步将卸载或删除应用本体。

不得删除：

- 用户导出到外部位置的备份；
- NF 数据根目录外的文件；
- 操作系统自身维护的数据；
- 其他应用数据。

Windows 安装版继续保留卸载器数据删除功能。

Windows 便携版清除退出后由用户删除便携目录。

macOS 清除退出后由用户删除 `NetraFlow.app`。

Ubuntu/Linux 清除退出后由用户删除 AppImage。

## 破坏性退出控制

最高级清除开始后，应用必须进入明确的破坏性关闭状态。

必须阻止：

- 窗口关闭时保存状态；
- 应用退出时重新写设置；
- WebContents 卸载时写入；
- 自动保存；
- 自动快照；
- 后台定时任务；
- 重新生成默认文件；
- 新的持久化 IPC；
- 删除后重新创建 `userdata` 或 `runtime`。

不得出现：

```text
数据被删除
→ 普通退出逻辑执行
→ 设置或默认数据重新生成
```

普通退出和清除后的退出必须明确区分。

## 权限与安全边界

NetraFlow 不得使用提权、隐藏或危险系统操作。

严禁：

```text
sudo
root
UAC 提权
管理员权限
pkexec
cmd.exe
PowerShell
Bash
sh
osascript
隐藏清理脚本
静默命令行清理
后台隐藏清理程序
修改系统保护设置
权限绕过
权限提升
```

应用功能只能使用：

- 当前用户普通权限；
- Electron 官方 API；
- Node.js 文件系统 API；
- 当前用户拥有的标准应用目录。

不得为了提高删除成功率申请更高权限。

## Shell 与外部进程限制

应用不得通过 Shell 完成清理、安装、修复或卸载。

禁止在应用代码中使用：

```text
exec
execFile
spawn cmd.exe
spawn powershell.exe
spawn bash
spawn sh
rm -rf
del
rmdir
Remove-Item
sudo
pkexec
osascript
```

清除数据必须使用 Electron 和 Node.js 标准 API。

不得创建隐藏临时脚本或命令文件。

## 窗口行为

Windows：

- 保留无边框窗口；
- 保留自绘最小化、最大化和关闭按钮；
- 保留现有拖拽和交互；
- 不为跨平台统一撤销 Windows 视觉设计。

macOS：

- 使用原生 traffic lights；
- 可保留 NF 自定义顶部内容区域；
- 不自行模拟红黄绿按钮；
- 正确处理 Dock、激活、关闭窗口和全屏。

Ubuntu/Linux：

- 第一版优先系统原生窗口控制；
- 可保留 NF 自定义内容标题区域；
- 不为不同 Linux 桌面环境绘制复杂仿系统按钮。

通用：

- 内容视觉可以统一；
- 外部窗口形状、阴影、圆角和全屏交给操作系统；
- 拖拽区域内交互控件必须不可拖拽；
- 不通过 CSS 强行模拟整个系统窗口外壳。

## 文件名和路径大小写

- import 路径必须与真实文件名大小写完全一致。
- 图片、字体、图标和资源路径必须大小写一致。
- 渲染进程和 Electron/Node 代码都应启用 TypeScript 大小写检查。
- 不依赖 Windows 大小写不敏感行为。
- Ubuntu/Linux 构建作为实际验证环境。
- 不需要增加复杂自定义扫描器。

## 移动端边界

当前不开发移动端。

不得：

- 创建 React Native；
- 创建 Capacitor；
- 创建移动端目录；
- 迁移 monorepo；
- 引入只为未来移动端准备的框架；
- 提前实现移动端文件系统；
- 提前实现移动端导航；
- 提前实现移动端安全存储。

可以保持：

- 业务规则尽量不依赖 Electron；
- 数据 schema 不依赖窗口；
- 校验逻辑保持纯 TypeScript；
- 加密格式不依赖桌面窗口；
- 数据迁移与 UI 分离；
- IPC 只作为桌面传输层。

不得为未来不确定需求增加当前复杂度。

## 依赖和构建克制性

- 未经批准不得升级依赖。
- 有 lockfile 时优先使用 `npm ci`。
- 无明确依赖变更不得修改 `package-lock.json`。
- 不擅自增加分析统计。
- 不擅自增加遥测。
- 不擅自增加后台服务。
- 不擅自增加开机启动。
- 不擅自增加系统托盘常驻。
- 不擅自增加自动更新。
- 不擅自增加系统级集成。
- 不擅自增加网络请求。
- 不擅自引入新权限。

## AI 修改前检查

修改前必须：

1. 阅读现有实现；
2. 确认 Windows 当前行为；
3. 确认必须保持的兼容行为；
4. 说明最小修改范围；
5. 检查平台差异；
6. 检查数据目录影响；
7. 检查删除路径安全；
8. 检查权限要求；
9. 检查 Shell 或外部命令；
10. 检查数据格式影响；
11. 检查用户导出文件影响；
12. 运行已有类型检查、测试和构建。

优先修复根因，不进行无关重构。

## AI 完成报告

任务完成后必须报告：

- 修改文件；
- 修改目的；
- 是否修改数据格式；
- 是否修改数据目录；
- 是否修改窗口行为；
- 是否修改构建配置；
- 是否增加依赖；
- 是否修改 lockfile；
- 执行的检查；
- 通过和失败情况；
- 当前 Git 状态；
- 是否创建本地 commit；
- 是否执行远端操作。

必须明确说明：

- 是否执行 `git push`；
- 是否创建远端分支；
- 是否设置 upstream；
- 是否创建 Pull Request；
- 是否创建或推送 tag；
- 是否创建 Release；
- 是否修改 GitHub 仓库。

未经授权，上述远端操作必须全部为“未执行”。

## 最终原则

NetraFlow 多平台扩展必须坚持：

> 一套业务代码、轻量平台适配、保留 Windows 兼容、尊重平台原生行为、数据集中管理、无提权、无静默命令、不依赖下次启动清除、未经允许绝不推送。

违反以上原则时，停止执行并报告，不得自行绕过。
